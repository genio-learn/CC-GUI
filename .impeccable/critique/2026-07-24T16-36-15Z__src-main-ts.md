## Design Specificity Verdict

**Strongly specific — this shell is authored for *this* product, not reskinned from a dashboard kit.** The evidence is in the state vocabulary, not the layout: the liveness dot *pulses* only when running (`style.css:1299`), waiting and hibernated sessions swap the circle for a `?` and `☾` glyph rather than another colored dot (`main.ts:1745–1762`), and one `--state-color` drives the dot, the sub-line chip, and the accent bar in lockstep (`style.css:293–314`). The "N waiting on you" pill, project-hashed identity colors (`--proj-0..7`), the cascade stack with a project-colored left spine, and terminal tabs whose active state fills `bg-base` with a 2px accent *top*-border are all product-specific choices.

What's category-interchangeable: the two-line list row, the `GROUP BY` segmented control, and the titlebar segment — standard patterns, competently executed. Net: the *composition* is generic; the *state language* is the brand — exactly as DESIGN.md intends ("brand lives in precise detail").

**Deterministic scan:** the detector ran clean on `index.html` — exit 0, zero findings. That is a thin signal, not a clearance: `index.html` is the only static markup, and the entire shell is DOM-built at runtime in `main.ts`, which a markup-only detector cannot see. Raw source counts corroborate the token discipline: **29 hex literals, all inside `:root` token definitions** (none scattered in components); **48 `color-mix()` usages** (tints built off tokens, as the contract requires); **7 `rgba()` literals** — 5 in shadow-token definitions, and only **2 scattered in component rules** (`style.css:786`, `794`), both plain black shadow/scrim. So the "no raw hex / no hardcoded rgba" contract is essentially honored; the two strays are cosmetic black overlays, not theme-breaking color.

**Visual overlays:** none. Browser visualization was skipped — CC-GUI is a Tauri app whose frontend needs the Rust backend to render, and no browser/screenshot tool is exposed this session. There is no user-visible overlay; findings rest on source + CSS read as rendered.

## Overall Impression

This is a mature, confident internal tool that already clears the bar most apps never reach: it has a real point of view (the themeable "working shell"), and the design contract is enforced in *code*, not just documented — `sessionStateKey` literally probes its own glyph output so the dot, chip, and bar can never drift (`main.ts:1785–1792`). The density holds, the token discipline holds, and nothing is decorative.

The single biggest opportunity is a gap between the stated north star and the built default: **"what needs you" is the reason this app exists, but in the default grouping it's carried by one titlebar integer and a scatter of 9px dots.** The signal deserves more weight than the chrome around it. Right behind that: the shell *looks* native but doesn't yet *behave* native for keyboard-only and assistive-tech users — session rows are plain divs with no roles.

## What's Working

1. **State-color lockstep is real, not aspirational.** `sessionStateKey` derives the state by probing `applyStatusGlyph`'s own output (`main.ts:1785–1792`), so the dot, sub-line chip, accent bar, and titlebar count are structurally incapable of drifting apart. This is the DESIGN.md contract made mechanical.
2. **Glyph substitution beats color-only encoding — in two places.** `?` for waiting and `☾` for hibernated (`main.ts:1747,1761`) mean those states survive a grayscale screenshot. It's the one spot the shell escapes hue-only signalling, and it's the right instinct.
3. **Density discipline is literal.** 6px row padding, an 18px reserved sub-line height so hover actions don't reflow rows below (`style.css:1201–1204`), and a reserved 2px border on tabs/rows so selection never shifts layout (`style.css:2866`). "Quiet frame that never jumps" is built, not just claimed.

## Priority Issues

**[P1] The north-star signal is under-weighted everywhere except one integer.**
- **Why it matters:** DESIGN's north star is literally "what needs you, and what's in progress." In the *default* Sections/Projects grouping there is no attention cluster at all — waiting sessions scatter among projects and the only per-row cue is a 9px dot. Even in Status grouping, the `Needs you` tier header renders with the *identical* `.project-header` style (dim, uppercase, 11px) as `Parked` (`main.ts:3010–3017`). The most important cluster has zero visual elevation. A 20+ session maintainer must eyeball every dot.
- **Fix:** Warning-tint + heavier weight on the `Needs you` header; make the titlebar attention pill click-to-filter; consider a pinned attention cluster that appears in *all* groupings, not only Status.
- **Suggested command:** `$impeccable bolder`

**[P1] Session rows have no accessibility semantics.**
- **Why it matters:** Rows are `<div class="session-row">` with click handlers — `main.ts` has **zero** `role`, `aria-*`, or `tabindex` (verified). The keyboard cursor is just `selectedId`, reached only via a bespoke `Cmd+Opt+↑/↓` chord; rows aren't in Tab order and expose nothing to assistive tech. "Full keyboard operability for every action" is a *binding* product constraint, and the custom titlebar + non-semantic rows trade looking-native for behaving-native.
- **Fix:** `role="listbox"`/`role="option"`, `tabindex`, `aria-current`/`aria-selected` on the cursor row, `aria-live="polite"` on the attention pill. Audit the several `outline: none` inputs for a visible focus replacement.
- **Suggested command:** `$impeccable harden`

**[P2] The mono-for-data rule is broken on the two most-scanned data fields.**
- **Why it matters:** Branch names (`.row-main .meta`, `style.css:1220`) and the PR badge `#123` (`.pr-badge`, `style.css:1678`) both inherit the sans UI face (verified — neither sets `font-family`). `.stack-name` correctly uses mono, so the rule exists and is applied unevenly. Sans branch names read as prose and lose column alignment — the exact fields a power user scans fastest.
- **Fix:** `font-family: var(--font-mono)` on `.row-main .meta` and `.pr-badge`.
- **Suggested command:** `$impeccable typeset`

**[P2] Four of six liveness states are color-only at 9px.**
- **Why it matters:** running / finished / idle / stopped are distinguished purely by hue (green / warning / cyan / dim) on a 9px dot (`style.css:1251–1266`); only waiting and hibernated carry a glyph. finished (yellow) vs running (green) vs idle (cyan) are hue-only for a color-blind user, and the running *pulse* is the sole non-color cue. The primary at-a-glance scan signal fails ~8% of users and every grayscale context — and finished/waiting, the two states that actually "need me," are the hardest to separate.
- **Fix:** Lean on the sub-line word chip as the guaranteed disambiguator (already present) and ensure it never ellipsises away; or add subtle shape variance to the dot per state.
- **Suggested command:** `$impeccable colorize`

**[P3] Raw backend error strings leak into toasts.**
- **Why it matters:** `toast(\`delete failed: ${e}\`, "error")` (`main.ts:1902,1966`) surfaces raw backend text to the user. Heuristic 9 scored 2 largely on this.
- **Fix:** Map to a human sentence; keep the raw string in a hover/details affordance.
- **Suggested command:** `$impeccable clarify`

## Persona Red Flags

**Alex (power user):** branch names in sans break fast column scanning (P2); the right-click session menu carries 7+ actions (Attach, shell, review, rename, editor, PR, lifecycle) with no separators or grouping — one decision point well over the ≤4 working-memory limit; the row comment chip reads "Comments" with no count because `SessionRow` carries only `has_pending_comments` (`main.ts:2055–2058`) — Alex wants the number.

**Sam (a11y / keyboard):** div rows with no `role`/`tabindex`/`aria` (P1); rows aren't Tab-reachable, only via a bespoke chord; multiple `outline: none` on inputs; state encoded by color at 9px (P2). The shell looks native but does not behave native for AT users.

**Maya (20+ concurrent-session maintainer):** in the default grouping there is no "which needs me" cluster — she must manually switch to Status grouping, where `Needs you` still isn't visually prioritized (P1). Per-row signal is one 9px dot, and finished/waiting are both yellow-family — the exact two states that need her are the hardest to separate at scale.

## Minor Observations

- A purpose-named `--attention` peach token exists and is used elsewhere (`style.css:3628+`), but the "waiting on you" pill and finished rows use `--warning` instead. The attention *signal* doesn't use the attention *token* — mild incoherence, not a bug.
- `renderStatusTiers` clusters by project in snapshot order (`main.ts:3035`); with churn, project sub-groups within "Needs you" can reorder between ticks.
- The zero-projects onboarding hero (`index.html:42–73`) is polished and specific; the zero-sessions-but-has-projects and 100-session states have no comparable authored treatment — the sidebar just grows a scrollbar.
- `#placeholder` copy "Select a session to attach." is static; it doesn't distinguish "nothing selected" from "selected session has no live terminal."

## Questions to Consider

1. If "what needs you" is the north star, why is it guaranteed visible only as a single titlebar integer — and absent as a cluster in the *default* grouping?
2. The system is themeable and mono-disciplined, so why do branch names and PR ids silently fall back to the sans face? Is the mono rule enforced anywhere, or is it convention?
3. At 20+ sessions, is a 9px color dot enough resolution — or should the sidebar auto-sort attention-first rather than making the user pick Status grouping?
4. Is "native" measured in pixels, or in behaving like a real desktop app for keyboard-only and AT users — where this shell currently falls short?
